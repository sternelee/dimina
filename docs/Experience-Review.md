# 项目长期执行规则

## 一、执行经验总结

### 1. 不要在错误层级修问题

多次纠偏集中在同一模式：问题表现出现在 demo、业务组件或页面上，但真正应修的位置是框架、编译器、SDK、native 容器或平台适配层。

问题做法：

- 为了让 demo 临时可用而修改 demo 或业务代码。
- 在业务组件里添加补丁式调用，比如手动触发 observer、额外设置延时、额外传入适配参数。
- 看到页面白屏、样式异常、按钮无效后直接改页面逻辑，没有先判断是框架、渲染、编译产物、native 容器还是业务层问题。

正确做法：

- 先确认用户指定的修复层级。用户多次明确要求“不要修改业务代码”“不要修改 demo 代码”“从框架层解决”“SDK 侧实现”。
- 对框架类问题，追踪到编译产物、运行时、双线程消息、native bridge、组件生命周期或官方实现，而不是在业务层打补丁。
- 如果业务侧加一行能证明问题，也只能把它当作定位线索，最终要把能力补回框架或 SDK。

教训：

- “能修好当前页面”不等于“修对了”。修复层级错误会引入更隐蔽的回归，也不符合用户偏好的工程质量。

### 2. 严禁硬编码作为长期方案

高频问题包括硬编码文件路径、平台目录、字段名、延时、固定高度、固定偏移和特定项目逻辑。

问题做法：

- 硬编码 `/dd`、`/wx`、特定项目包名、特定字段、固定 1.5s、两个 animation frame、固定高度、固定 centerOffset。
- 为单一 case 添加特判，没有证明它是框架语义的一部分。

正确做法：

- 从最终编译产物、配置、运行时状态、API 返回、平台能力或标准规范里推导值。
- 对 UI 高度、位置、动画，优先使用布局、内容尺寸、CSS/native 动画能力、safe area、平台组件能力，而不是固定数值。
- 对路径和依赖，参考微信小程序官方文档、Vite 迁移文档等权威来源。

教训：

- 硬编码会让一个 case 通过，同时让相邻 case 失败。用户更接受慢一点的根因修复，不接受“看起来可用”的脆弱补丁。

### 3. 对齐微信小程序官方实现

多次正确方向来自“对齐微信小程序生命周期”“参考官方文档”。

问题做法：

- 凭直觉重写小程序行为、组件 observer、property observer、生命周期顺序、native API 行为。
- 没有查看底层源码就假设 DRN/Harmony/Android/iOS 的实现一致。

正确做法：

- 对小程序框架行为，优先查微信官方文档。
- 对 DRN/Harmony/Android/iOS 差异，优先查平台适配层、native SDK、DNavigation、Map SDK、Mpx runtime、编译产物。
- 对开源库升级或协议行为，优先查官方 migration guide、规范文档和现有测试。

教训：

- 用户的工程理念不是“近似实现”，而是“语义对齐”。尤其是框架项目，兼容语义比局部代码简洁更重要。

### 4. 不用延时掩盖时序问题

典型问题包括父子组件事件注册与消费顺序、service/render 双线程、测量时机过早、页面二次进入不稳定、微任务无法跨线程保证顺序。

问题做法：

- 用 `setTimeout`、多次 `nextTick`、animation frame 或延后注册 observer 来绕开时序。
- 未区分 service 线程生命周期与 render 线程真实渲染完成。
- 只验证第一次进入页面，没有验证重复进入、快速点击、收起再展开等不稳定路径。

正确做法：

- 画清楚事件产生、注册、消费、渲染、测量的顺序链路。
- 用统一前缀日志定位 service 与 render 两侧的事件顺序，确认实际先后关系。
- 将生命周期顺序对齐微信小程序或平台真实实现。
- 对不稳定问题要重复验证，至少覆盖首次进入、二次进入、快速点击、展开/收起循环。

教训：

- “延迟后正常”通常只是暴露了时序问题，不是修复。最终方案要让事件顺序本身成立。

### 5. 验证必须覆盖相邻回归

记录中多次出现“Pop 正常了但 Calendar 又报错”“Image 正常但 Icon 不显示”“不报错但 body 为空”“样式修了但按钮被遮挡”等回归。

问题做法：

- 只验证当前报错消失。
- 没有跑相关示例、相邻组件、历史回归点。
- 忽略“无报错但无内容”的视觉失败。

正确做法：

- 修一个框架问题时，至少验证原始复现、相邻组件、此前被同一链路影响的页面。
- UI/渲染问题必须看最终画面或 DOM/截图，不以构建成功或无异常作为完成。
- 用户提供“实际/预期”或截图时，按截图做视觉验收。

教训：

- 对框架和跨端代码，最危险的结果是“当前 case 通过，其他 case 被改坏”。验证范围要按影响链路扩大。

### 6. 日志要可定位，也要及时移除

用户多次接受“加日志帮助排查”，也多次要求“移除多余调试日志”。

正确做法：

- 需要定位时加统一前缀、结构化字段、关键时序点，便于用户回传过滤。
- 日志要说明问题链路，不要铺满无关输出。
- 修复后移除临时日志，保留必要的异常日志或业务可观测点。

教训：

- 好日志是定位工具，不是最终实现的一部分。

### 7. 复杂问题先读当前状态，不凭记忆

会话中大量任务依赖当前未提交改动、最近提交、PR diff、用户本地日志、编译产物路径、node_modules 源码、oh_modules 源码。

正确做法：

- 开始前先看 `git status`、相关文件、最近提交、编译产物、日志路径。
- 遇到用户说“之前是正常的”“最近某个提交改坏了”，优先用 git history/diff/bisect 思路定位。
- 不回滚用户改动，除非用户明确要求。

教训：

- 用户常常已经提供了定位线索。Agent 的价值是把线索系统化验证，而不是重新猜。

## 二、用户偏好与理念档案

### 工程理念

- 偏好根因修复，不接受“能跑就行”的补丁。
- 偏好在正确层级修复：框架问题修框架，SDK 问题修 SDK，native 容器问题修 native，业务逻辑问题才改业务。
- 偏好语义对齐：微信小程序、官方文档、平台源码、已有项目规范优先于自行推测。
- 偏好通用机制：用框架能力、配置、编译器机制、运行时协议解决一类问题，而不是特判一个页面。
- 偏好保持现有行为：修新问题不能破坏旧页面、旧动画、旧组件、旧平台。
- 偏好明确证据：日志、截图、编译产物、源码链路、diff、测试结果要能支撑结论。

### UI 与视觉偏好

- 重视像素级对齐：图标、文字、按钮、卡片、地图控件要垂直居中、间距稳定、位置准确。
- 不接受遮挡和截断：底部安全区、横条、列表、卡片、按钮、两行文案都不能互相遮挡或被裁切。
- 偏好与原设计一致：按钮尺寸、圆角、图片圆角、环境切换页、标题栏、菜单样式要延续已有视觉语言。
- 不喜欢无谓变形：按钮“不要圆边”并不等于完全直角，应该保持原有图片或系统既定圆角。
- 偏好紧凑清晰的操作面板：功能菜单不应过高；能一行完成的按钮不要堆成复杂页面；不必要按钮应移除。
- 偏好真实信息结构：AppId、复制链接、重新进入路径、关闭小程序等功能要符合真实微信小程序或容器行为。
- 偏好动画完整、顺滑、不过慢：按钮要完成升起，卡片展开/收起不能跳变，父容器高度和位置变化要丝滑。
- 偏好触感与反馈克制：按钮可增加震动反馈；错误提示快速重复触发时只显示一次并延迟消失。
- 偏好图片自然呈现：图片不应被错误压缩或收缩；必要时撑开布局比压扁图片更好。
- 偏好地图控件语义正确：定位点居中优先考虑 `bestViewPadding` 等地图能力；归位、搜停车点、规划路线按钮应在列表或底部卡片上方，不被遮挡。

### 产品与交互原则

- 功能应贴近真实业务路径，不为测试便利创建与真实体验不一致的入口。
- 文案要统一、明确、可本地化；中英文、日文、韩文等语言都不能导致布局破坏。
- 状态反馈要节制：连接中置灰、未连接提示合并、关闭与隐藏要区分。
- 交互不应依赖用户“刚好慢点点”：快速点击、重复进入、切换环境、展开收起都应稳定。
- 工具和调试入口要符合现有产品结构，不要额外制造不一致页面。

## 三、未来 Agent 可直接遵循的规则

1. 开工前先确认修复层级；用户说不要改业务/demo 时，绝不改业务/demo 作为最终方案。
2. 先读当前状态：`git status`、相关源码、编译产物、日志、最近提交、用户指定路径和现有实现。
3. 不用硬编码作为最终方案；路径、尺寸、延时、字段、平台差异都要从配置、标准、运行时或源码推导。
4. 框架行为优先对齐官方文档、平台源码和已有测试；不要凭直觉重写兼容语义。
5. 遇到时序问题，追踪事件注册、消费、渲染、测量的完整链路；不要用 `setTimeout`、多次 `nextTick` 或固定 animation frame 掩盖。
6. DRN/Harmony/Mpx/小程序相关问题要同时考虑 service/render 双线程、native bridge、平台适配层和编译产物。
7. UI 任务必须以最终视觉为准：检查截图、DOM/布局、safe area、遮挡、截断、对齐、动画完整性。
8. 修复后验证原始复现和相邻回归点；框架改动必须扩大验证到同链路组件或页面。
9. 如果不能运行完整验证，必须明确说明未验证项和残余风险，不能把“代码看起来合理”当作完成。
10. 定位困难时可以加日志，但日志要统一前缀、字段清晰；完成后移除临时调试日志。
11. 对用户提供的“实际/预期”“我怀疑”“是不是”要当作高价值线索，先验证再反驳。
12. 遇到“之前正常、最近坏了”，优先查 git diff、历史提交、回归范围和被同一改动影响的页面。
13. 产品交互要尊重现有风格和真实业务路径，不额外创造与真实路径不一致的按钮页、测试页或入口。
14. 文案和布局要考虑多语言、两行文本、长词、按钮尺寸和垂直居中，不能截断或错位。
15. 动画优化优先减少测量和父容器重排，尽量使用 CSS/native transform、稳定布局和平台组件能力。
16. 地图、导航、扫码、蓝牙、压缩、相机等平台能力问题，先查 SDK/平台源码与参数语义，再决定上层适配。
17. 长任务要阶段性落盘，防止上下文压缩、网络错误或进程中断导致结论丢失。
18. 最终回复要说清改了什么、验证了什么、哪些没有验证；不要堆砌过程，也不要隐藏失败。

## 四、默认执行模板

后续 Agent 会话遇到代码任务时，默认按下面顺序执行：

1. 定位：读取用户指定路径、相关日志、当前 diff、最近提交、权威参考。
2. 判断层级：确认是业务、框架、编译器、SDK、native 容器还是平台适配问题。
3. 复现或构造最小证据：能运行就运行，不能运行就用源码、产物、日志建立证据链。
4. 实施：小范围修改，优先通用机制，禁止硬编码和临时延时。
5. 回归：验证原始问题、相邻路径、视觉状态和平台差异。
6. 收尾：移除临时日志，报告验证结果与残余风险。
